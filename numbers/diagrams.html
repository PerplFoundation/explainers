LNS is wider than PNS (40 vs 32 bits) because lot counts can be vastly larger than prices — a trader holding 500 billion of a near-zero-price token needs more range than any realistic price. All four types are chosen to pack tightly into EVM storage words, minimising gas cost per operation.

BTC example (1 + 5 = 6)
price = $95,000.1 → PNS = 950,001
size = 2.50000 BTC → LNS = 250,000
raw product = 950,001 × 250,000
             = 237,500,250,000
÷ 106 = $237,500.25 USDC ✓
MON example (6 + 0 = 6)
price = $0.400000 → PNS = 400,000
size = 5,000 MON → LNS = 5,000
raw product = 400,000 × 5,000
             = 2,000,000,000
÷ 106 = $2,000.00 USDC ✓
If price_dec + size_dec were < 6, precision would be wasted — the product would always end in zeros. If > 6, the product would have too many decimal places and fractional USDC would be silently dropped, causing rounding errors in PnL. Setting both live perps to exactly 6 is the only lossless choice.

42× headroom to ONS ceiling. If MON exceeds ~$16, basePricePNS shifts up.
basePricePNS is a fixed offset stored per-perp. The orderbook represents all prices as ONS values (ticks above this base). When the market moves outside the ONS window, a privileged transaction updates basePricePNS, shifting the window. For BTC with 1 price decimal, the window is wide enough that this almost never happens. For MON with 6 price decimals, a large rally would require a window shift.
